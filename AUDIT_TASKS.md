## PHASE 1: CRITICAL — Fix Before Anything Else
> These tasks address exploitable vulnerabilities or showstopper defects.
> Estimated: 4 hours

- [x] TASK-001 — Add CSRF protection to all mutating API routes
  - Domain: Security
  - Severity: CRITICAL
  - Files: `utils/api/guards.ts`, all `app/api/*/route.ts`
  - What to do: The current `assertSameOrigin` check relies on the Origin header which can be spoofed in legacy browsers or certain proxy setups. Implement double-submit cookie pattern or use Next.js built-in CSRF protection via server actions. Generate a CSRF token on login and validate it on all POST/PUT/DELETE endpoints server-side.
  - Done when: All mutating endpoints reject requests without a valid CSRF token.

- [x] TASK-002 — Implement server-side password policy (minimum strength)
  - Domain: Security
  - Severity: CRITICAL
  - Files: `app/account/signup/page.tsx`, `utils/useAuth.js`
  - What to do: Add server-side password validation on signup. Minimum 8 characters, at least one letter and one number. Currently only client-side emptiness check exists. Validate in `useAuth.js` signUpWithCredentials or add an API validation step before Supabase auth call.
  - Done when: Passwords shorter than 8 chars or without mixed character types are rejected server-side with a clear error message.

- [x] TASK-003 — Validate file upload MIME type by content magic bytes, not just header
  - Domain: Security
  - Severity: CRITICAL
  - Files: `app/api/upload/route.ts`
  - What to do: The current `ALLOWED_TYPES.has(file.type)` checks the Content-Type header sent by the browser/client, which is trivially spoofed. Add magic-byte validation (e.g. using `file-type` npm package) to verify the actual file content matches the declared type before storing in Supabase Storage.
  - Done when: Uploads with mismatched content-type header vs actual file content are rejected.
